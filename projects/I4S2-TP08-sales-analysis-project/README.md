## Module-by-Module Explanation

### 1. `main.py` — Entry Point

```python
from src.cleaner import clean
from src.loader import create_spark_session, load_csv
from src.analyzer import analyze
from src.visualizer import plot_sales_by_store, plot_top_cities

DATA_PATH = "data/Retail_Transactions_Dataset.csv"

def main() -> None:
    spark = create_spark_session()
    try:
        raw = load_csv(spark, DATA_PATH)
        df = clean(raw)
        result = analyze(df)
        plot_sales_by_store(result.store_sales_pdf)
        plot_top_cities(result.city_sales_pdf)
    finally:
        spark.stop()
```

**What it does:**

1. **`create_spark_session()`** — creates the Spark entry point (the driver). This is the "remote control" for the Spark cluster (even though we run locally).
2. **`load_csv(...)`** — reads the CSV into a `DataFrame`. No computation happens yet (lazy evaluation).
3. **`clean(raw)`** — replaces the string `"None"` in the `Promotion` column with actual SQL `null`, then drops rows with nulls in key numeric columns.
4. **`analyze(df)`** — triggers computation: total sales, top store, average sale. Returns an `AnalysisResult` dataclass containing both numbers and small pandas DataFrames for plotting.
5. **`plot_sales_by_store(...)`** and **`plot_top_cities(...)`** — save PNG charts to `reports/`.
6. **`finally: spark.stop()`** — always shuts down the SparkSession, even if an exception occurs. This releases JVM memory and worker threads.

The `try/finally` pattern is important for Spark applications — if you forget `spark.stop()`, the JVM may keep running in the background.

### 2. `src/loader.py` — Spark Session & CSV Loading

```python
from pathlib import Path
from pyspark.sql import DataFrame, SparkSession


def create_spark_session(app_name: str = "SalesAnalysis") -> SparkSession:
    return (
        SparkSession.builder
        .appName(app_name)
        .master("local[*]")
        .config("spark.sql.shuffle.partitions", "8")
        .getOrCreate()
    )


def load_csv(spark: SparkSession, csv_path: str) -> DataFrame:
    path = Path(csv_path).resolve()
    if not path.exists():
        msg = f"CSV file not found at {path}"
        raise FileNotFoundError(msg)
    return spark.read.csv(str(path), header=True, inferSchema=True)
```

#### `create_spark_session()`

| Setting | Meaning |
|---------|---------|
| `appName("SalesAnalysis")` | Identifies the job in the Spark UI (`localhost:4040`) |
| `master("local[*]")` | Runs Spark locally using **all available CPU cores** (`*`). In a real cluster this would be `"yarn"` or `"spark://master:7077"`. |
| `config("spark.sql.shuffle.partitions", "8")` | Limits the number of shuffle partitions to 8. The Spark default is **200** — designed for large clusters. For a local 1M-row dataset with few group keys, 8 is faster and uses less memory. |
| `.getOrCreate()` | Reuses an existing `SparkSession` if one exists, avoiding conflicts in Jupyter/notebook environments. |

**What is a SparkSession?** It is the unified entry point that replaces `SparkContext`, `SQLContext`, and `HiveContext` from earlier Spark versions. Through it you access:
- DataFrames / SQL (`spark.read`, `spark.sql`)
- Configuration (`spark.conf`)
- Catalog of tables (`spark.catalog`)

#### `load_csv()`

- `Path(csv_path).resolve()` — converts a relative path like `"data/file.csv"` to an absolute path so Spark's JVM can find it.
- `spark.read.csv(..., header=True, inferSchema=True)`:
  - `header=True` — first row is column names
  - `inferSchema=True` — Spark scans the data to detect integer, double, boolean, and string columns automatically. Without this, **every column would be read as a string**.
  - Returns a **lazy** `DataFrame` — no data has been read yet.

**Spark DataFrame vs Pandas DataFrame:**
- Pandas holds all data in memory on one machine
- Spark holds a **logical plan** (recipe) and splits actual data across partitions
- Spark DataFrames are **lazy** — they only materialize when an action runs

### 3. `src/cleaner.py` — Missing Value Handling

```python
from pyspark.sql import DataFrame
from pyspark.sql.functions import col, when


def clean(df: DataFrame) -> DataFrame:
    initial_count = df.count()
    cleaned = df.withColumn(
        "Promotion",
        when(col("Promotion") == "None", None).otherwise(col("Promotion"))
    ).dropna(subset=["Total_Cost", "Total_Items"])

    final_count = cleaned.count()
    removed = initial_count - final_count
    print(f"  Rows before: {initial_count:,}")
    print(f"  Rows after:  {final_count:,}")
    print(f"  Removed:     {removed:,}")
    return cleaned
```

#### Why a `clean()` step?

The dataset uses the literal string `"None"` in the `Promotion` column (333K rows) to mean "no promotion." This is a common issue in real-world CSVs — missing data is encoded as a placeholder string instead of being empty or `NULL`. We need to convert it so Spark understands these are truly missing values.

#### Line-by-line

| Expression | What it does |
|------------|-------------|
| `df.count()` | **Action** — triggers a full scan to count rows. Forces Spark to actually read the data. |
| `df.withColumn("Promotion", ...)` | Returns a new DataFrame with the `Promotion` column replaced. Spark DataFrames are **immutable** — transformations always create new objects. |
| `when(col("Promotion") == "None", None)` | `when(condition, value)` is Spark's equivalent of SQL's `CASE WHEN`. If the column equals the string `"None"`, return Python's `None` (which becomes SQL `null`). |
| `.otherwise(col("Promotion"))` | Otherwise keep the original value. |
| `.dropna(subset=["Total_Cost", "Total_Items"])` | Drop rows where `Total_Cost` or `Total_Items` is `null`. We only check numeric columns because those would break arithmetic. |

#### Two `.count()` calls — why?

Each `.count()` triggers a separate Spark job:
1. First count — scans all rows to get `initial_count`
2. After `withColumn`/`dropna` — scans again for `final_count`

**Important:** Spark transformations are lazy. Without the first `.count()`, no work happens at all. The print statement shows the user the impact of cleaning.

### 4. `src/analyzer.py` — Aggregation Logic

```python
from dataclasses import dataclass
import pandas as pd
from pyspark.sql import DataFrame
from pyspark.sql.functions import avg, col, desc, sum


@dataclass
class AnalysisResult:
    total_sales: float
    top_store: str
    top_store_sales: float
    avg_sales: float
    store_sales_pdf: pd.DataFrame
    city_sales_pdf: pd.DataFrame


def analyze(df: DataFrame) -> AnalysisResult:
    total_sales = df.select(sum("Total_Cost")).collect()[0][0] or 0.0

    top_store_row = (
        df.groupBy("Store_Type")
        .agg(sum("Total_Cost").alias("total"))
        .orderBy(desc("total"))
        .first()
    )
    top_store = top_store_row["Store_Type"] if top_store_row else "N/A"
    top_store_sales = top_store_row["total"] if top_store_row else 0.0

    avg_sales = df.select(avg("Total_Cost")).collect()[0][0] or 0.0

    store_sales = (
        df.groupBy("Store_Type")
        .agg(sum("Total_Cost").alias("total"))
        .orderBy(desc("total"))
        .toPandas()
    )

    city_sales = (
        df.groupBy("City")
        .agg(sum("Total_Cost").alias("total"))
        .orderBy(desc("total"))
        .limit(10)
        .toPandas()
    )

    print(f"\nResults:")
    print(f"  Total sales:      ${total_sales:,.2f}")
    print(f"  Top-selling store: {top_store} (${top_store_sales:,.2f})")
    print(f"  Average sale:     ${avg_sales:,.2f}")

    return AnalysisResult(
        total_sales=total_sales,
        top_store=top_store,
        top_store_sales=top_store_sales,
        avg_sales=avg_sales,
        store_sales_pdf=store_sales,
        city_sales_pdf=city_sales,
    )
```

#### `AnalysisResult` dataclass

A **dataclass** is a Python class that auto-generates `__init__`, `__repr__`, and `__eq__` from type-annotated fields. We use it here to bundle all results into a single returned object so `main.py` has a clean contract with `analyzer.py`.

#### Three computations

**Total Sales:**
```python
df.select(sum("Total_Cost")).collect()[0][0]
```
- `df.select(sum("Total_Cost"))` — creates a DataFrame with one column (`sum(Total_Cost)`) and one row
- `.collect()` — **action** that sends all data to the driver as a Python list of `Row` objects → `[Row(sum(Total_Cost)=52455220.4)]`
- `[0][0]` — first row, first column → `52455220.4`
- `or 0.0` — safety if result is `None` (empty dataset)

**Top-Selling Store:**
```python
df.groupBy("Store_Type")
  .agg(sum("Total_Cost").alias("total"))
  .orderBy(desc("total"))
  .first()
```
- `groupBy("Store_Type")` — partition rows by store type (shuffle happens here)
- `.agg(sum("Total_Cost").alias("total"))` — within each group, sum the costs
- `.orderBy(desc("total"))` — sort descending by the sum
- `.first()` — action that returns only the top row (optimized — Spark doesn't need to sort the entire dataset, it can use a top-N algorithm)

**Average Sales:**
```python
df.select(avg("Total_Cost")).collect()[0][0]
```
Identical pattern to total, but uses `avg()` instead of `sum()`.

#### `.toPandas()` — when and why

```python
store_sales = df.groupBy("Store_Type").agg(...).orderBy(...).toPandas()
```

`.toPandas()` converts a Spark DataFrame to a **pandas** DataFrame by collecting all data to the driver. This is **only safe** when the result is small (here, 6-7 store types). Doing `.toPandas()` on the full 1M-row dataset would crash the driver with an out-of-memory error.

We use pandas here because seaborn/matplotlib work natively with pandas DataFrames, not Spark DataFrames.

### 5. `src/visualizer.py` — Plotting Results

```python
from pathlib import Path
import matplotlib.pyplot as plt
import pandas as pd
import seaborn as sns

REPORTS_DIR = Path(__file__).resolve().parent.parent / "reports"


def plot_sales_by_store(store_sales: pd.DataFrame) -> Path:
    REPORTS_DIR.mkdir(parents=True, exist_ok=True)
    path = REPORTS_DIR / "sales_by_store.png"

    plt.figure(figsize=(10, 6))
    sns.barplot(
        data=store_sales,
        x="Store_Type",
        y="total",
        hue="Store_Type",
        palette="viridis",
        legend=False,
    )
    plt.title("Total Sales by Store Type")
    plt.xlabel("Store Type")
    plt.ylabel("Total Sales ($)")
    plt.xticks(rotation=45)
    plt.tight_layout()
    plt.savefig(str(path))
    plt.close()

    print(f"  Plot saved: {path}")
    return path


def plot_top_cities(city_sales: pd.DataFrame) -> Path:
    REPORTS_DIR.mkdir(parents=True, exist_ok=True)
    path = REPORTS_DIR / "top_cities.png"

    plt.figure(figsize=(10, 6))
    sns.barplot(
        data=city_sales.sort_values("total"),
        x="total",
        y="City",
        hue="City",
        palette="magma",
        legend=False,
    )
    plt.title("Top 10 Cities by Total Sales")
    plt.xlabel("Total Sales ($)")
    plt.ylabel("City")
    plt.tight_layout()
    plt.savefig(str(path))
    plt.close()

    print(f"  Plot saved: {path}")
    return path
```

#### Why `plt.close()`?

Matplotlib keeps figures in memory until explicitly closed. Without `plt.close()`, running the script repeatedly in a notebook or loop would leak memory. Each function creates a figure, saves it, then closes it.

#### Why `hue` with `legend=False`?

Seaborn v0.14+ requires `hue` when using `palette`. By setting `hue` to the same categorical column and disabling the redundant legend, we get colored bars without a useless color key.

#### `Path(__file__).resolve()`

`__file__` is the path to `visualizer.py`. `.resolve()` makes it absolute. `parent.parent` goes up two levels to the project root, then we append `reports/`. This makes the script **location-independent** — it works no matter which directory you run from.

## How Spark Distributes the Work

This is the most important concept to understand. Here is exactly what happens when you run `main.py`:

### 1. Partitioning

When Spark reads a CSV, it splits the file into **partitions** — contiguous chunks of rows. With `master("local[*]")` on an 8-core machine, Spark typically creates 8-16 partitions (depending on file size and Hadoop input format).

```text
CSV file (1M rows)
├── Partition 0 — rows 0–124,999   →  Executor core 1
├── Partition 1 — rows 125,000–249,999 →  Executor core 2
├── Partition 2 — rows 250,000–374,999 →  Executor core 3
├── ...
└── Partition 7 — rows 875,000–999,999 →  Executor core 8
```

**Why partitioning matters:** Each partition is processed independently and in parallel. The total time ≈ time_for_one_partition × (total_partitions / available_cores). With 8 partitions and 8 cores, all partitions run simultaneously.

### 2. Lazy Evaluation & the DAG

Spark does **not** process data as you call methods. Instead, every transformation (`withColumn`, `groupBy`, `select`) adds a node to a **Directed Acyclic Graph (DAG)** — a logical execution plan. Only **actions** (`.count()`, `.collect()`, `.first()`, `.toPandas()`, `.save()`) trigger actual computation.

**Why lazy?** It lets Spark's optimizer (Catalyst):
- **Pipeline** narrow transformations (filter → select → withColumn) into a single stage without shuffling
- **Reorder** operations for efficiency (push filters closer to the data source)
- **Prune** columns that are never used in the final result

### 3. Stages & Shuffle Boundaries

A **wide dependency** (also called a shuffle) occurs when data must be redistributed across partitions — for example, `groupBy("Store_Type")` because rows for "Pharmacy" might be in any partition.

Spark splits the DAG at shuffle boundaries into **stages**:

```text
Stage 0 (narrow)                           Stage 1 (wide)
┌────────────────────┐                     ┌─────────────────────┐
│ Read CSV           │                     │ Shuffle read        │
│ └─ Partition 0     │   ─── shuffle ──→   │ └─ "Pharmacy" rows  │
│ └─ Partition 1     │                     │ └─ "Supermarket"    │
│ └─ ...             │                     │ └─ ...              │
│ withColumn(...)    │                     │ group + aggregate   │
│ dropna(...)        │                     │ orderBy             │
└────────────────────┘                     └─────────────────────┘
```

- **Stage 0**: Read CSV, clean nulls — no data crosses partition boundaries
- **Stage 1**: Hash-partition rows by `Store_Type`, then aggregate within each group

### 4. Task Execution

Each partition in each stage becomes a **task**. Tasks are the unit of parallelism:

```text
Stage 0 (8 tasks)          Stage 1 (8 tasks)
[task p0] [task p1]        [task s0] [task s1]
[task p2] [task p3]  ──→   [task s2] [task s3]
[task p4] [task p5]        [task s4] [task s5]
[task p6] [task p7]        [task s6] [task s7]
   8 cores, parallel          8 cores, parallel
```

In local mode, all tasks run in separate threads within the same JVM. On a real cluster, tasks run on different machines.

### 5. The Shuffle — What Actually Happens

When `groupBy("Store_Type")` executes:

1. **Map side:** Each task computes a **local** aggregate per key (e.g., "Pharmacy → $1,234.56"). It writes the result to **shuffle files** on disk.
2. **Shuffle transfer:** The `HashPartitioner` hashes each key: `hash("Pharmacy") % num_partitions`. Executors fetch the files they need from other executors (or other threads in local mode).
3. **Reduce side:** Each task reads all rows for its assigned keys and computes the **final** aggregate (sum of all partial sums).

**Why shuffles are expensive:**
- Disk I/O (writing/reading shuffle files)
- Network transfer (in cluster mode)
- Serialization/deserialization overhead
- They are the #1 performance bottleneck in Spark

Our config `spark.sql.shuffle.partitions=8` controls how many partitions the data is shuffled into. Too few = large partitions, memory pressure. Too many = small files, scheduling overhead.

### 6. Fault Tolerance via Lineage

Every Spark DataFrame remembers the sequence of transformations that created it — its **lineage**.

```text
df.rdd.toDebugString()
(8) MapPartitionsRDD[7] at ... []
 |  MapPartitionsRDD[6] at ... []
 |  MapPartitionsRDD[5] at ... []
 |  MapPartitionsRDD[4] at ... []
 |  MapPartitionsRDD[3] at ... []
 |  FileScanRDD[2] at ... []
```

If a task fails (e.g., executor crash, OOM), Spark:
1. Identifies which partitions were lost
2. Finds the **ancestor RDD** where those partitions are still available (checkpoint, HDFS, or source)
3. Re-executes only the lost partition's tasks by replaying the lineage

**No manual checkpointing needed** — this is automatic.

### 7. Comparison with Pandas

| Aspect | Pandas | PySpark |
|--------|--------|---------|
| **Data location** | Single machine RAM | Distributed across executors (or local threads) |
| **Processing** | Single-core (most ops) | Multi-core / multi-node |
| **Lazy evaluation** | No — eager | Yes — DAG built, then executed |
| **Max dataset size** | RAM of one machine | RAM of cluster (hundreds of TB) |
| **Fault tolerance** | None — crash = restart | Lineage-based recomputation |
| **Shuffle** | N/A (single process) | Explicit, expensive phase |
| **API style** | `.groupby().sum()` | `.groupBy().agg(sum())` |
| **Best for** | <10 GB, interactive exploration | >10 GB, scheduled pipelines |

**Why use PySpark for this 1M-row exercise?** Even though 1M rows fits in pandas, the exercise teaches partitioning, lazy evaluation, shuffle behavior, and the DAG — concepts that scale to billions of rows. The same code, with minor config changes, runs on a 100-node cluster.
